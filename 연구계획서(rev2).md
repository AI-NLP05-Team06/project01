# 연구 계획서

## AI 서비스 데이터 파이프라인 구축

> 본 프로젝트는 예금보험공사와 금융안심포털에 분산된 웹 데이터를 수집·정제하여, 향후 RAG 기반 예솜24 챗봇이 신뢰도 높은 답변을 생성할 수 있도록 하는 데이터 파이프라인의 프로토타입을 구축하는 것을 목표로 한다.
>
> 본 과제는 단순히 웹페이지를 수집하는 문제가 아니라, **사용자 질문에 필요한 공식 정보를 선별하고, 원문의 구조와 출처를 보존하며, 변경사항을 다시 반영할 수 있는 형태로 데이터를 관리하는 문제**로 해석한다.

---

## 1. 프로젝트 개요

### 1.1 프로젝트 목표

- 6개 업무 도메인 페이지를 결정론적으로 텍스트로 변환한다.
- 페이지별 LLM 추론에 활용하기 위한 요약 메타데이터를 생성한다.
- 출처 URL, 업무 카테고리, 첨부파일 등 메타데이터를 태깅한다.
- 콘텐츠 변경·삭제 발생 시 관리자가 트리거링하여 재수집 및 재적재할 수 있는 파이프라인을 구성한다.
- RAG 평가에 활용할 질의-답변 테스트셋을 구축한다.

### 1.2 대상 업무

1. 예금자보호제도
2. 예금보험금 안내
3. 고객 미수령금 신청
4. 착오송금 반환 신청
5. 채무조정 안내
6. 은닉재산 신고

### 1.3 데이터 소스

- 예금보험공사: `https://www.kdic.or.kr`
- 금융안심포털: `https://fins.kdic.or.kr`

두 사이트 중 하나를 일괄적인 주 데이터 소스로 정하지 않는다. 업무별 사용자 질문에 필요한 공식 정보가 어느 사이트에 존재하는지 확인한 뒤, 적절한 페이지를 수집 대상으로 선정한다.

## 2. 과제 해석

### 2.1 RAG 이전 단계의 데이터 품질 문제

본 프로젝트는 RAG 모델이나 챗봇 자체를 개발하는 과제가 아니라, **RAG가 신뢰할 수 있는 근거 데이터를 사용할 수 있도록 만드는 데이터 엔지니어링 과제**로 해석한다.

웹사이트는 사람이 메뉴를 따라 탐색하도록 구성되어 있지만, 사용자는 다음과 같이 자신의 상황을 중심으로 질문한다.

- “내 예금은 얼마까지 보호되나요?”
- “잘못 송금한 돈을 돌려받으려면 어떻게 해야 하나요?”
- “상속인이 예금보험금을 신청할 때 필요한 서류는 무엇인가요?”
- “채무조정 신청 대상과 절차를 알려주세요.”

따라서 웹페이지의 본문만 단순 수집하면 다음과 같은 문제가 발생할 수 있다.

- 같은 표현을 사용하는 다른 업무의 문서가 함께 검색될 수 있다.
- 제목, 소제목, 표, 목록과 본문의 관계가 사라질 수 있다.
- 첨부파일이나 관련 링크가 누락될 수 있다.
- 원문 출처를 역추적하기 어려울 수 있다.
- 기존에 수집한 데이터가 변경된 페이지와 함께 남을 수 있다.
- 수집한 데이터가 실제 질문에 답할 수 있는지 확인하기 어렵다.

이에 따라 본 프로젝트의 핵심을 다음 네 가지로 정의한다.

1. **재현 가능한 변환**  
   동일한 HTML 입력에는 동일한 텍스트가 생성되어야 한다.

2. **검색 가능한 구조화**  
   업무, 하위 카테고리, 출처, 첨부파일 등의 메타데이터를 통해 검색 범위를 구분할 수 있어야 한다.

3. **변경 대응 가능성**  
   관리자가 파이프라인을 다시 실행했을 때 신규·변경·삭제된 콘텐츠를 판별할 수 있어야 한다.

4. **평가 가능성**  
   공식 페이지에 근거한 질문·정답·출처 데이터를 통해 향후 RAG 검색 및 답변 품질을 평가할 수 있어야 한다.

### 2.2 질문 중심의 수집 범위

본 프로젝트에서는 사이트 전체를 무조건 수집하는 방식보다, **사용자가 실제로 물을 가능성이 높은 질문과 그 질문의 공식 근거가 되는 페이지를 연결하는 방식**을 우선한다.

페이지별로 다음 항목을 확인한다.

- 해당 페이지가 6개 업무 중 어느 업무에 속하는가
- 어떤 사용자 질문에 답하기 위한 근거가 되는가
- 다른 페이지와 내용이 중복되거나 연결되어 있는가
- 표, 첨부파일, 이미지에 핵심 정보가 존재하는가
- 정적 페이지인지 동적 조회 페이지인지
- 변경 가능성이 높은 정보인지

이를 통해 수집 대상을 `필수`, `검토`, `제외`로 구분하고, 일주일 동안에는 필수 페이지를 우선 처리한다.

## 3. 수행 범위

### 3.1 필수 구현 범위

본 프로젝트에서는 다음 기능이 하나의 흐름으로 동작하는 프로토타입을 구현한다.

```text
업무별 URL 목록
        ↓
원본 HTML 수집
        ↓
결정론적 HTML → 텍스트 변환
        ↓
페이지 요약 및 메타데이터 생성
        ↓
청킹 및 JSONL 저장
        ↓
관리자 트리거 기반 변경 판별
        ↓
질의-답변 테스트셋 구축
```

### 3.2 프로젝트 1에서 다루지 않는 범위

일주일의 개발 기간을 고려하여 다음 기능은 필수 구현 범위에서 제외한다.

- 실시간 또는 상시 웹페이지 모니터링
- 관리자용 웹 대시보드
- 모든 동적 페이지에 대한 브라우저 자동화
- 모든 이미지의 OCR 또는 멀티모달 자동 처리
- 벡터 데이터베이스와 실제 챗봇 서비스의 완전한 연동
- Parent-Child Retrieval, HyDE 등 고급 검색 기법
- 대규모 자동 평가 플랫폼
- 수백 건 이상의 평가 질문 구축

위 기능은 기본 파이프라인이 완성된 이후 확장 가능한 항목으로 관리한다.

## 4. 데이터 수집 및 변환 방법

### 4.1 URL 목록 구성

6개 업무별로 공식 페이지를 조사하고 다음 정보를 기록한다.

```yaml
source_url: 원본 페이지 URL
business_function: 소속 업무
sub_category: 업무 내 하위 분류
collection_priority: required / review / excluded
expected_question: 해당 페이지가 근거가 되는 대표 질문
```

프로젝트 기간에는 `required` 페이지를 우선 수집하며, `review` 페이지는 구현 진행 상황에 따라 추가한다.

### 4.2 원본 데이터 수집

기본 수집기는 `requests`를 이용하여 HTML을 가져오고 다음 정보를 함께 저장한다.

- 요청 URL
- 최종 URL
- HTTP 상태 코드
- 수집 시각
- 원본 HTML
- 페이지 내 첨부파일 링크
- 페이지 내 이미지 URL

정적 수집이 불가능한 페이지는 실패 원인을 기록하고, 프로젝트 1에서는 별도 처리 대상 목록으로 분리한다.

### 4.3 결정론적 HTML → 텍스트 변환

원문 변환에는 규칙 기반 파서를 사용한다. 동일한 HTML에 대해 동일한 결과를 얻기 위해 원문 생성 단계에서는 LLM을 사용하지 않는다.

변환 시 다음 구조를 최대한 보존한다.

- 페이지 제목
- 소제목
- 문단
- 순서형·비순서형 목록
- 표
- 본문 링크
- 첨부파일 이름과 URL

반대로 다음 영역은 제거한다.

- 공통 헤더
- 전체 메뉴
- 푸터
- 배너
- 반복되는 사이트 안내 문구

출력 형식은 Markdown을 기본으로 하며, 원본 HTML과 변환 결과를 함께 보관한다.

### 4.4 이미지 처리

일주일 동안 모든 이미지를 자동으로 텍스트화하지 않는다. 대신 이미지 URL, 대체 텍스트, 주변 문맥을 메타데이터로 저장한다.

핵심 절차나 조건이 이미지에만 존재하는 경우에는 다음 중 하나를 적용한다.

- 사람이 직접 내용을 확인하여 텍스트로 보완
- 제한된 수의 핵심 이미지만 멀티모달 모델로 설명 생성
- 프로젝트 한계 및 추가 처리 대상에 기록

## 5. 메타데이터 설계

### 5.1 페이지 메타데이터

```json
{
  "doc_id": "deposit_protection_001",
  "site_name": "예금보험공사",
  "source_url": "https://example.com/page",
  "title": "보호한도",
  "business_function": "예금자보호제도",
  "sub_category": "보호한도",
  "summary": "예금보험제도의 보호한도와 산정 기준을 안내하는 페이지",
  "collection_priority": "required",
  "content_type": "static_page",
  "collected_at": "2026-07-14T10:00:00",
  "content_hash": "sha256-value",
  "attachments": [],
  "images": []
}
```

### 5.2 필수 메타데이터

| 필드 | 목적 |
| --- | --- |
| `doc_id` | 페이지 고유 식별 |
| `source_url` | 원문 출처 추적 |
| `title` | 문서 제목 보존 |
| `business_function` | 6개 업무 간 검색 범위 구분 |
| `sub_category` | 업무 내 세부 주제 구분 |
| `summary` | 페이지의 핵심 내용 파악 |
| `collected_at` | 데이터 수집 시점 확인 |
| `content_hash` | 변경 여부 판별 |
| `attachments` | 첨부파일 이름과 URL 보존 |
| `images` | 이미지 URL과 주변 정보 보존 |

### 5.3 LLM 요약 메타데이터

LLM은 원문 변환이 아니라 **보조 메타데이터 생성**에 사용한다.

페이지별 변환 텍스트를 입력으로 사용하여 다음 정보를 생성한다.

- 1~2문장의 페이지 요약
- 페이지가 답변할 수 있는 대표 질문
- 핵심 키워드

모든 페이지에는 동일한 프롬프트와 출력 형식을 적용하여 결과를 일관되게 관리한다.

## 6. 청킹 및 적재 형식

프로젝트 1에서는 전체 흐름을 안정적으로 구현하기 위해 단순한 규칙 기반 청킹을 사용한다.

기본 원칙은 다음과 같다.

- 소제목 경계를 우선 고려한다.
- 하나의 표는 가능한 한 분리하지 않는다.
- 너무 긴 섹션만 일정 글자 수를 기준으로 추가 분할한다.
- 모든 청크는 원본 페이지를 역참조할 수 있어야 한다.

### 청크 데이터 예시

```json
{
  "chunk_id": "deposit_protection_001_chunk_002",
  "parent_doc_id": "deposit_protection_001",
  "business_function": "예금자보호제도",
  "sub_category": "보호한도",
  "section_title": "보호금액 산정 기준",
  "chunk_index": 2,
  "content": "예금보험금은 원금과 소정의 이자를 합하여...",
  "source_url": "https://example.com/page"
}
```

### 출력 파일

```text
data/
├── raw/
│   └── html/
├── parsed/
│   └── markdown/
├── processed/
│   ├── documents.jsonl
│   └── chunks.jsonl
└── evaluation/
    └── qa_dataset.jsonl
```

## 7. 관리자 트리거 기반 갱신

관리자가 명령어를 실행하면 전체 수집 또는 변경 확인을 수행할 수 있도록 구성한다.

```bash
python run_pipeline.py --mode full
python run_pipeline.py --mode update
```

### 7.1 전체 실행

`full` 모드에서는 등록된 전체 URL을 수집·변환하고 메타데이터와 청크를 생성한다.

### 7.2 업데이트 실행

`update` 모드에서는 기존 데이터와 현재 페이지를 비교한다.

```text
URL 목록 확인
    ↓
신규 URL 여부 확인
    ↓
기존 URL은 변환 본문의 해시 비교
    ↓
new / unchanged / updated / deleted 판별
    ↓
변경이 필요한 문서만 재처리
```

| 상태 | 처리 |
| --- | --- |
| `new` | 신규 수집·변환·적재 |
| `unchanged` | 기존 데이터 유지 |
| `updated` | 문서와 관련 청크 교체 |
| `deleted` | 비활성화 또는 적재 대상에서 제외 |
| `failed` | 오류 내용 기록 |

삭제 판별은 기존 URL 목록과 현재 관리 URL 목록의 차이를 기준으로 구현한다. 실제 사이트의 모든 삭제를 자동 탐지하는 기능은 프로젝트 1 범위에 포함하지 않는다.

## 8. 질의-답변 테스트셋

### 8.1 목적

테스트셋은 모델이 얼마나 자연스럽게 답변하는지를 평가하기보다, **사용자 질문에 필요한 공식 근거가 수집 데이터 안에 존재하는지 확인하기 위한 기준 데이터**로 사용한다.

### 8.2 구성 원칙

- 6개 업무가 모두 포함되어야 한다.
- 질문은 공식 페이지에서 답을 확인할 수 있어야 한다.
- 정답과 함께 근거 문서 ID와 출처 URL을 기록한다.
- 정의, 대상, 조건, 절차, 서류, 금액 등 주요 질문 유형을 포함한다.
- 프로젝트 기간에는 업무별 5개 이상, 전체 30~60개를 목표로 한다.

### 8.3 데이터 예시

```json
{
  "question_id": "qa_001",
  "business_function": "예금자보호제도",
  "question_type": "금액형",
  "question": "예금은 얼마까지 보호되나요?",
  "gold_answer": "공식 페이지를 기준으로 작성한 정답",
  "gold_doc_id": "deposit_protection_001",
  "source_url": "https://example.com/page",
  "answerable": true
}
```

## 9. 검증 방법

프로젝트 1에서는 복잡한 모델 성능 비교보다 파이프라인이 요구사항에 맞게 동작하는지 검증한다.

### 9.1 수집 검증

- 등록 URL 중 정상 수집된 페이지 비율
- 실패 페이지와 실패 원인
- 첨부파일 URL 보존 여부
- 원본 HTML 저장 여부

### 9.2 변환 검증

대표 페이지를 선정하여 다음 항목을 사람이 확인한다.

- 제목과 소제목이 보존되었는가
- 문단 순서가 원문과 일치하는가
- 목록과 표의 핵심 구조가 보존되었는가
- 헤더·푸터 등 불필요한 영역이 제거되었는가
- 동일 HTML을 다시 변환했을 때 동일한 결과가 생성되는가

### 9.3 메타데이터 검증

- 필수 필드 누락 여부
- 업무 분류 일치 여부
- 출처 URL 연결 여부
- 첨부파일과 이미지 정보 보존 여부
- 청크와 원본 문서 연결 여부

### 9.4 갱신 검증

테스트 HTML을 이용해 다음 상황을 재현한다.

1. 신규 페이지 추가
2. 기존 본문 수정
3. URL 목록에서 페이지 제거
4. 변경되지 않은 페이지 재실행

각 상황에서 상태가 `new`, `updated`, `deleted`, `unchanged`로 올바르게 판별되는지 확인한다.

### 9.5 테스트셋 검증

- 6개 업무별 질문 수 확인
- 질문과 정답의 공식 근거 확인
- `gold_doc_id`와 실제 수집 문서 연결 확인
- 출처 URL 누락 여부 확인

## 13. 예상 리스크 및 대응 방안

| 리스크 | 대응 방안 |
| --- | --- |
| 페이지별 HTML 구조가 다름 | 공통 파서를 우선 적용하고 예외 페이지는 사이트별 규칙 또는 실패 목록으로 관리 |
| 동적 페이지 수집 실패 | 정적 수집 가능 여부를 먼저 확인하고 동적 페이지는 별도 대상 목록으로 분리 |
| 표 구조 손실 | 대표 페이지를 기준으로 표 변환 결과를 직접 검수하고 필요한 규칙만 추가 |
| 핵심 정보가 이미지에만 존재 | 이미지 URL과 주변 문맥을 저장하고 핵심 이미지만 수작업 또는 제한적 멀티모달 처리 |
| 업무 간 유사 표현으로 인한 혼입 | `business_function`과 `sub_category`를 필수 메타데이터로 적용 |
| LLM 요약 결과의 형식 불일치 | 동일 프롬프트와 JSON 출력 형식을 적용하고 실패 결과는 재생성 |
| 변경되지 않은 페이지의 반복 처리 | 변환 본문의 해시를 비교하여 미변경 페이지는 재처리하지 않음 |
| 일주일 내 범위 초과 | 필수 URL과 필수 메타데이터를 우선하고 고급 검색·자동화 기능은 제외 |
| 평가 질문의 근거 부족 | 모든 질문에 `gold_doc_id`와 공식 출처 URL을 필수로 연결 |

```text
project/
├── src/
│   ├── crawler.py
│   ├── parser.py
│   ├── metadata.py
│   ├── chunker.py
│   └── run_pipeline.py

│   ├── urls/
│   ├── raw/html/
│   ├── parsed/markdown/
│   ├── processed/documents.jsonl
│   ├── processed/chunks.jsonl
│   └── evaluation/qa_dataset.jsonl
├── tests/
├── 연구계획서.md
└── README.md
```

최종 결과물은 다음 조건을 만족해야 한다.

1. 명령어를 통해 6개 업무의 데이터를 수집·변환할 수 있다.
2. 변환 결과에서 원본 출처와 업무 분류를 확인할 수 있다.
3. 관리자가 다시 실행했을 때 변경된 데이터만 구분할 수 있다.
4. 질의-답변 테스트셋의 질문과 근거 문서를 연결할 수 있다.
5. 다른 사용자가 README를 보고 동일한 파이프라인을 실행할 수 있다.

## 15. 기대 효과

본 프로젝트를 통해 웹페이지를 단순 저장하는 수준을 넘어, 향후 RAG 시스템이 검색·인용·평가에 활용할 수 있는 데이터 구조를 확보할 수 있다.

특히 다음 효과를 기대한다.

- 공식 출처 기반 답변을 위한 데이터 기반 확보
- 6개 업무 간 검색 범위 구분
- 원문 구조와 첨부파일 정보 보존
- 변경 콘텐츠의 선택적 갱신 가능
- RAG 검색 및 답변 품질을 반복 검증할 수 있는 기준 데이터 확보

본 프로젝트의 성공 기준은 모든 기능을 운영 수준으로 완성하는 것이 아니라, **수집·변환·메타데이터·갱신·평가 데이터가 하나의 재실행 가능한 흐름으로 연결되는 프로토타입을 완성하는 것**이다.
