# 연구 계획서

## AI 서비스 데이터 파이프라인 구축

> 본 프로젝트는 예금보험공사와 금융안심포털에 분산된 웹 데이터를 수집·정제하여, 향후 RAG 기반 예솜24 챗봇이 신뢰도 높은 답변을 생성할 수 있도록 하는 데이터 파이프라인 구축을 목표로 한다.
>
> 단순히 웹페이지를 크롤링하는 것이 아니라, **어떤 데이터를 수집해야 하는지 판단하고, 검색에 적합한 형태로 구조화하며, 변경사항을 반영할 수 있는 데이터셋을 설계하는 과정**에 중점을 둔다.

---

## 1. Problem Definition

### 1.1 문제점: 비즈니스 니즈

예금보험공사와 금융안심포털에는 예금자보호제도, 예금보험금, 착오송금 반환, 채무조정 등 다양한 업무 안내 정보가 존재한다.

그러나 현재 웹페이지는 일반 사용자가 직접 탐색하는 구조를 기준으로 만들어져 있으며, AI 검색이나 RAG 기반 챗봇이 활용하기 위한 데이터 구조로 설계되어 있지 않다.

주요 문제는 다음과 같다.

* 6개 업무 관련 정보가 두 개의 웹사이트와 여러 하위 메뉴에 분산되어 있다.
* 동일하거나 유사한 표현이 여러 업무 페이지에서 반복된다.
* 페이지의 제목, 메뉴 구조, 브레드크럼, 첨부파일, 이미지 등의 문맥 정보가 본문과 분리되어 있다.
* 표, 첨부파일 링크, 이미지 설명 등이 단순 텍스트 변환 과정에서 유실될 수 있다.
* 정책 변경이나 페이지 수정·삭제 시 기존 적재 데이터와 최신 웹사이트 내용이 달라질 수 있다.
* RAG 시스템의 검색 및 답변 성능을 평가하기 위한 표준 질의-답변 데이터가 부족하다.

따라서 비즈니스 요구사항은 다음과 같이 정의할 수 있다.

> 예금보험공사 관련 웹페이지를 AI가 검색하고 인용할 수 있는 데이터 형태로 변환하고, 변경사항이 발생했을 때 다시 수집·적재할 수 있는 데이터 파이프라인을 구축한다.

### 1.2 Task Reformulation

#### 초기 접근 방법

초기에는 프로젝트를 다음과 같이 단순하게 해석할 수 있었다.

> 예금보험공사 웹사이트의 필요한 페이지를 크롤링하여 본문 텍스트를 저장한다.

이 접근은 웹페이지의 텍스트를 확보하는 데는 충분할 수 있지만, 실제 RAG 시스템에서 활용하기에는 다음과 같은 한계가 있다.

* 어떤 페이지를 수집해야 하는지 판단 기준이 없다.
* 페이지 간 업무 구분이 없어 다른 업무의 유사 문서가 함께 검색될 수 있다.
* 표, 이미지, 첨부파일, 메뉴 경로 등의 정보가 손실될 수 있다.
* 동일한 내용의 중복 페이지와 FAQ가 함께 적재될 수 있다.
* 페이지 변경 및 삭제 여부를 추적하기 어렵다.
* 수집 데이터가 실제 질문에 답변 가능한지 평가할 방법이 없다.

즉, 단순 크롤링만으로는 다음 질문에 답할 수 없다.

* 어떤 페이지가 RAG에 반드시 필요한가?
* 어떤 페이지는 중복 또는 검색 노이즈를 유발하는가?
* 검색 결과가 어떤 업무에 속하는지 어떻게 구분할 것인가?
* 웹페이지 내용이 변경되면 기존 데이터는 어떻게 갱신할 것인가?
* 구축한 데이터가 실제 질문에 답할 수 있는지 어떻게 검증할 것인가?

#### 현재 접근 방법

우리 팀은 본 프로젝트를 단순한 크롤링 문제가 아니라 다음 네 가지 하위 문제로 재해석하였다.

1. **수집 범위 정의**

   * 6개 업무별로 사용자 질문 대응에 필요한 페이지를 선별한다.
   * 필수 수집 페이지와 추가 분석이 필요한 페이지를 구분한다.

2. **결정론적 문서 변환**

   * 웹페이지 원본 HTML을 보존한다.
   * 동일한 입력 HTML은 항상 동일한 텍스트 결과로 변환되도록 한다.
   * 표, 소제목, 링크, 첨부파일 등의 구조를 최대한 유지한다.

3. **검색용 메타데이터 설계**

   * 업무, 하위 분류, 출처 URL, 브레드크럼, 첨부파일, 수집일 등의 정보를 구조화한다.
   * 향후 계층 필터링과 Parent-Child Retrieval이 가능하도록 설계한다.

4. **갱신 및 평가 체계 구축**

   * 페이지의 변경·삭제를 감지하여 재수집할 수 있는 구조를 만든다.
   * 기존 FAQ와 수집 문서를 기반으로 질의-답변-출처 URL 평가셋을 구축한다.

### 1.3 Task 설계

본 프로젝트의 최종 Task는 다음과 같이 정의한다.

> **예금보험공사 6개 업무 도메인의 웹페이지를 수집하고, RAG 검색에 활용할 수 있는 텍스트·메타데이터·평가 데이터로 변환하며, 변경사항을 반영할 수 있는 재수집 파이프라인을 구축하는 문제**

세부 Task는 다음과 같다.

#### Task 1. 수집 대상 URL 정의

* 예금보험공사와 금융안심포털의 사이트맵 및 메뉴 구조 분석
* 6개 업무별 관련 페이지 분류
* 필수 수집, 분석 필요, 선택 수집으로 구분
* 페이지별 수집 필요 이유와 예상 사용자 질문 정의

#### Task 2. HTML 원문 수집

* URL별 원본 HTML 저장
* 요청 시각, 응답 상태, 최종 URL 등 수집 정보 기록
* 동일 페이지 재수집 시 변경 여부 비교가 가능하도록 원본 보존

#### Task 3. HTML → 텍스트 변환

* 헤더, 푸터, 메뉴 등 불필요한 영역 제거
* 제목, 소제목, 문단, 목록, 표의 구조 유지
* 첨부파일과 링크 정보 보존
* LLM에 의존하지 않는 결정론적 변환 방식 적용

#### Task 4. 메타데이터 생성

* 페이지별 업무 및 하위 분류 태깅
* URL, 제목, 브레드크럼, 수집일자, 첨부파일 정보 저장
* 청크와 원본 페이지의 관계를 나타내는 문서 ID 설계
* 필요 시 LLM을 활용한 페이지 요약 메타데이터 생성

#### Task 5. 청킹 및 적재 데이터 생성

* 초기에는 글자 수 또는 토큰 수 기반 청킹 적용
* 소제목과 표 경계를 고려하여 문맥 단절 최소화
* 청크별 메타데이터 연결
* 향후 RAG 벡터DB에 적재할 수 있는 JSON 또는 JSONL 형식으로 변환

#### Task 6. 평가용 QA 데이터셋 구축

* 기존 예솜24 FAQ와 수집 페이지를 기반으로 질의 생성
* 정답, 근거 문서, 출처 URL을 함께 저장
* 검색 성능과 답변 성능을 반복 평가할 수 있는 기준 데이터로 활용

#### Task 7. 갱신 파이프라인 구축

* 관리자 트리거 방식으로 재수집 실행
* 기존 HTML과 신규 HTML의 변경 여부 비교
* 변경 페이지 재파싱 및 재적재
* 삭제 페이지의 비활성화 또는 제거 처리

### 1.4 이 문제가 중요한 이유

#### 1. 사용자가 원하는 것은 페이지가 아니라 답변이다

사용자는 웹사이트 메뉴 구조를 직접 탐색하기보다 다음과 같이 자연어로 질문한다.

* “내 예금은 얼마까지 보호돼?”
* “저축은행도 보호 대상이야?”
* “착오송금 반환 신청은 어떻게 해?”
* “상속인이 예금보험금을 신청하려면 무슨 서류가 필요해?”

RAG 챗봇이 이러한 질문에 정확하게 답하려면 단순한 페이지 텍스트가 아니라, 질문과 연결될 수 있는 구조화된 데이터가 필요하다.

#### 2. 검색 오류는 데이터 구조에서 발생할 수 있다

6개 업무에서는 다음과 같은 표현이 반복적으로 등장한다.

* 신청 방법
* 신청 기한
* 제출 서류
* 수수료
* 지급 대상
* 처리 절차

전체 데이터를 하나의 평면적인 벡터 공간에서 검색하면 다른 업무의 유사한 청크가 검색될 수 있다.

예를 들어 사용자가 착오송금 반환 신청 기한을 물었는데, 채무조정 신청 기한 문서가 함께 검색될 수 있다.

따라서 수집 단계에서부터 업무별 계층 구조와 필터링에 필요한 메타데이터를 만들어야 한다.

#### 3. 최신성이 중요한 공공정보 데이터다

예금보호한도, 신청 조건, 대상 금융회사, 지급 대상 회사 등은 변경될 수 있다.

과거 정보와 최신 정보가 동시에 벡터DB에 남으면 RAG 시스템이 오래된 문서를 근거로 잘못된 답변을 생성할 수 있다.

따라서 변경·삭제 감지와 재수집 기능은 선택 기능이 아니라 데이터 신뢰성을 위한 핵심 요소다.

#### 4. 데이터 품질이 RAG 성능의 상한을 결정한다

검색 모델이나 LLM의 성능이 높더라도 다음 문제가 있으면 정확한 답변을 생성하기 어렵다.

* 필요한 페이지가 수집되지 않음
* 표의 행과 열 관계가 깨짐
* 첨부파일 링크가 누락됨
* 페이지의 업무 분류가 없음
* 청크가 너무 짧거나 길어 문맥이 손실됨
* 이전 버전의 문서가 삭제되지 않음

따라서 본 프로젝트에서는 모델보다 먼저 데이터 품질과 파이프라인 구조를 설계한다.

## 2. Background & Baseline

### 2.1 Related Approaches

#### 단순 정적 크롤링

`requests`와 `BeautifulSoup`을 이용해 웹페이지 HTML을 수집하고 본문 텍스트를 추출하는 방식이다.

구현은 간단하지만 다음과 같은 문제가 있다.

* JavaScript 기반 동적 콘텐츠 수집이 어렵다.
* 표와 계층 구조가 쉽게 손실된다.
* 사이트별 본문 선택자를 개별적으로 작성해야 한다.
* 페이지 변경 및 삭제 관리가 별도로 필요하다.

#### 브라우저 자동화 기반 크롤링

Selenium 또는 Playwright를 이용해 JavaScript가 실행된 이후의 페이지를 수집하는 방식이다.

동적 페이지 처리에는 유리하지만 다음 단점이 있다.

* 실행 속도가 느리다.
* 자원 사용량이 크다.
* 브라우저 환경에 따른 재현성 문제가 발생할 수 있다.

#### 범용 문서 추출 라이브러리

Trafilatura, Readability, Unstructured 등의 도구를 이용하여 웹페이지 본문을 추출하는 방식이다.

범용성이 높지만, 공공기관 사이트의 표·첨부파일·메뉴 구조를 정확하게 보존하는지 별도 검증이 필요하다.

#### LLM 기반 웹페이지 변환

웹페이지를 LLM에 입력하여 중요 내용을 추출하거나 요약하는 방식이다.

비정형 문서 처리에는 유리하지만 다음 문제가 있다.

* 동일한 입력에 대해 결과가 달라질 수 있다.
* 원문 내용이 누락되거나 재해석될 수 있다.
* 비용과 처리 시간이 증가한다.
* 수집 원문 보존이라는 목적과 맞지 않는다.

따라서 LLM은 원문 변환이 아니라 요약 메타데이터 생성 등 보조 작업에 제한적으로 활용한다.

### 2.2 이번 프로젝트의 Baseline

본 프로젝트의 초기 baseline은 다음과 같이 설정한다.

#### Baseline Pipeline

```text
사이트맵 URL 수집
        ↓
requests 기반 HTML 수집
        ↓
BeautifulSoup 기반 본문 추출
        ↓
제목·문단·목록·표 텍스트 변환
        ↓
글자 수 기반 단순 청킹
        ↓
페이지 및 청크 메타데이터 저장
        ↓
JSONL 형식 출력
```

#### Baseline 메타데이터

초기에는 다음 필드를 중심으로 구성한다.

```yaml
doc_id: 원본 페이지 식별자
chunk_id: 청크 식별자
source_url: 출처 URL
title: 페이지 제목
business_function: 6개 업무 중 소속 업무
sub_category: 업무 내 하위 분류
breadcrumb: 웹사이트 메뉴 경로
collected_at: 수집 시각
content_hash: 본문 변경 확인용 해시
parent_doc_id: 청크가 속한 원본 문서 ID
attachments: 첨부파일 목록
```

#### Baseline 선정 이유

* 구현 난도가 지나치게 높지 않아 초기 프로토타입을 빠르게 검증할 수 있다.
* 원문 HTML과 변환 텍스트를 직접 비교할 수 있다.
* 오류 발생 지점을 수집, 파싱, 청킹, 메타데이터 단계별로 구분할 수 있다.
* 향후 동적 크롤링, 구조 기반 청킹, 자동 요약 등 고도화 기능과 비교하기 쉽다.

## 3. Proposed Method

### 3.1 핵심 아이디어

우리 팀의 핵심 접근은 다음과 같다.

> **업무별 수집 범위를 먼저 정의하고, 웹페이지 구조를 보존한 결정론적 변환과 계층 메타데이터를 결합하여, 검색 노이즈를 줄이고 재수집 가능한 RAG 데이터 파이프라인을 구축한다.**

단순히 많은 페이지를 수집하는 것보다 다음 세 가지를 우선한다.

1. **필요한 데이터만 수집**
2. **원문의 구조와 출처를 보존**
3. **검색과 갱신에 필요한 메타데이터를 함께 생성**

### 3.2 Baseline 대비 개선점

| 구분    | Baseline                | Proposed                                                 |
| ----- | ----------------------- | -------------------------------------------------------- |
| 수집 범위 | 사이트맵에 존재하는 관련 페이지 일괄 수집 | 업무별 필수·분석 필요·선택 수집 기준 적용                                 |
| 본문 추출 | 단순 HTML 태그 제거           | 제목·소제목·표·목록·첨부파일 구조 보존                                   |
| 업무 구분 | URL 또는 폴더 수준 구분         | `business_function`, `sub_category`, `breadcrumb` 기반 계층화 |
| 청킹    | 글자 수 기준 단순 분할           | 제목·소제목·표 경계를 고려한 구조 기반 청킹                                |
| 중복 처리 | 별도 처리 없음                | URL, 본문 해시, 의미 중복 후보를 기준으로 관리                            |
| 변경 관리 | 전체 재수집                  | 변경·추가·삭제 페이지 판별 후 선택 갱신                                  |
| 평가    | 수집 건수 중심                | QA 테스트셋 기반 검색 가능성 및 근거 정확성 검증                            |
| 활용 목적 | 텍스트 저장                  | RAG 검색·인용·평가에 직접 활용 가능한 데이터셋 생성                          |

### 3.3 Dataset & Preprocessing

#### 데이터 소스

본 프로젝트에서 사용하는 데이터는 다음 두 개의 공개 웹사이트에서 수집한다.

* 예금보험공사

  * `https://www.kdic.or.kr`
* 금융안심포털

  * `https://fins.kdic.or.kr`

#### 대상 업무

수집 대상은 예솜24 챗봇의 주요 6개 업무를 기준으로 한다.

5. 채무조정 안내
6. 은닉재산 신고

#### 초기 사이트 분석 결과

| 업무         |     필수 |  분석 필요 |     합계 |
| ---------- | -----: | -----: | -----: |
| 예금자보호제도    |      5 |      2 |      7 |
| 예금보험금 안내   |      3 |      1 |      4 |
| 고객 미수령금 신청 |      3 |      1 |      4 |
| 착오송금 반환 신청 |     17 |      5 |     22 |
| 채무조정 안내    |      3 |      5 |      8 |
| 은닉재산 신고    |      2 |      2 |      4 |

> 위 결과는 프로젝트 착수 전 초기 분석이며, 실제 수집 및 검증 과정에서 대상 페이지가 추가되거나 제외될 수 있다.

#### 수집 여부 판단 기준

각 페이지는 다음 기준으로 평가한다.

* 실제 사용자 질문에 답변하기 위해 필요한가
* 기존 페이지와 내용이 중복되는가
* 최신 정책이나 예외 조건을 포함하는가
* RAG 검색 결과에 노이즈를 유발할 가능성이 있는가
* 정적 페이지인지 동적 조회 데이터인지
* 유지보수 및 갱신 비용이 과도하지 않은가
* QA 평가셋 생성에 활용할 수 있는가

#### 전처리 과정

```text
원본 HTML 저장
        ↓
인코딩 및 응답 상태 확인
        ↓
본문 영역 식별
        ↓
메뉴·배너·푸터 등 불필요 영역 제거
        ↓
제목·소제목·문단·목록 추출
        ↓
표를 행·열 관계가 유지되는 텍스트로 변환
        ↓
링크·첨부파일·이미지 정보 추출
        ↓
공백 및 불필요한 반복 문자열 정규화
        ↓
페이지 단위 텍스트 생성
        ↓
청크 분할 및 메타데이터 연결
```

#### 원본 보존 원칙

다음 세 종류의 데이터를 함께 보관한다.

```text
raw/
└── 원본 HTML

parsed/
└── 구조화 텍스트 또는 Markdown

processed/
└── 청크 및 메타데이터 JSONL
```

이 구조를 통해 변환 오류가 발견됐을 때 원본 HTML부터 다시 처리할 수 있도록 한다.

### 3.4 메타데이터 설계

#### 페이지 메타데이터 예시

```json
{
  "doc_id": "deposit_protection_limit_001",
  "source_url": "https://example.com/page",
  "title": "보호한도",
  "business_function": "예금자보호제도",
  "sub_category": "보호한도",
  "breadcrumb": [
    "예금자보호제도",
    "보호한도"
  ],
  "collection_priority": "required",
  "content_type": "static_page",
  "collected_at": "YYYY-MM-DDTHH:MM:SS",
  "content_hash": "sha256-value",
  "attachments": [],
  "images": [],
  "summary": "금융회사별 예금자 1인당 보호한도를 설명하는 페이지"
}
```

#### 청크 메타데이터 예시

```json
{
  "chunk_id": "deposit_protection_limit_001_chunk_003",
  "parent_doc_id": "deposit_protection_limit_001",
  "business_function": "예금자보호제도",
  "sub_category": "보호한도",
  "section_title": "보호금액 산정 기준",
  "chunk_index": 3,
  "content": "예금보험금은 원금과 소정의 이자를 합하여...",
  "source_url": "https://example.com/page"
}
```

#### 계층 메타데이터가 필요한 이유

전체 데이터를 하나의 검색 공간에서 조회하면, 서로 다른 업무에서 동일한 표현을 포함한 청크가 함께 검색될 수 있다.

따라서 다음과 같은 검색 흐름을 지원할 수 있어야 한다.

```text
사용자 질문
    ↓
질문이 속한 업무 분류
    ↓
business_function 필터 적용
    ↓
해당 업무 내 child 청크 검색
    ↓
parent_doc_id를 통해 원본 페이지 문맥 확장
    ↓
답변 생성 및 출처 제시
```

본 프로젝트에서는 검색 모델을 직접 구현하지 않지만, 위 구조가 가능하도록 데이터와 메타데이터를 제공한다.

### 3.5 갱신 파이프라인 설계

#### 변경 감지 방식

재수집 시 기존 데이터와 신규 데이터를 비교한다.

```text
기존 URL 목록
      +
신규 URL 목록
      ↓
추가 / 유지 / 삭제 URL 분류
```

유지되는 URL은 본문 해시를 비교한다.

```text
기존 content_hash
        +
신규 content_hash
        ↓
동일: 재적재 생략
변경: 재파싱 및 재적재
```

#### 상태 구분 예시

| 상태          | 의미             | 처리              |
| ----------- | -------------- | --------------- |
| `new`       | 신규 페이지         | 수집·파싱·적재        |
| `unchanged` | 내용 변화 없음       | 기존 데이터 유지       |
| `updated`   | 본문 또는 메타데이터 변경 | 기존 청크 교체        |
| `deleted`   | 사이트에서 삭제됨      | 비활성화 또는 벡터DB 제거 |
| `failed`    | 수집 또는 파싱 실패    | 오류 로그 및 재시도     |

#### 관리자 트리거 방식

초기 프로토타입은 자동 스케줄링보다 관리자가 직접 실행하는 구조로 설계한다.

```bash
python run_pipeline.py --mode full
```

```bash
python run_pipeline.py --mode update
```

```bash
python run_pipeline.py --business-function "예금자보호제도"
```

이를 통해 전체 재수집, 변경분 수집, 특정 업무 재수집을 구분할 수 있도록 한다.

### 3.6 QA 평가셋 설계

#### 평가셋 구성

각 샘플은 다음 정보를 포함한다.

```json
{
  "question_id": "qa_001",
  "business_function": "예금자보호제도",
  "question": "내 예금은 얼마까지 보호돼?",
  "answer": "금융회사별 예금자 1인당 원금과 이자를 합하여 1억 원까지 보호됩니다.",
  "source_url": "https://example.com/page",
  "source_doc_id": "deposit_protection_limit_001",
  "question_type": "fact",
  "difficulty": "easy"
}
```

#### 질문 유형

* 개념 설명형
* 조건 확인형
* 절차 안내형
* 구비서류형
* 대상 여부 확인형
* 예외 조건형
* 최신 정책형
* 특정 금융회사 조회형

#### 평가셋 활용 목적

* 필요한 문서가 검색되는지 확인
* 출처 URL이 올바르게 연결되는지 확인
* 업무 분류 메타데이터가 적절한지 확인
* 청킹으로 인해 정답 근거가 분리되지 않았는지 확인
* 향후 RAG 검색 방식 또는 모델 변경 시 공통 비교 기준으로 사용

---

### 3.7 효과가 있을 것으로 예상하는 이유

#### H1. 업무 계층 메타데이터는 타 업무 문서 혼입을 줄일 것이다

6개 업무별로 반복되는 표현이 많으므로, `business_function` 기반 검색 범위 제한은 검색 정확도를 높일 것으로 예상한다.

#### H2. 구조 보존형 파싱은 표와 조건문의 정보 손실을 줄일 것이다

공공기관 안내 페이지에서는 표의 행과 열, 소제목과 본문 관계가 중요하다.

단순 태그 제거보다 구조를 유지한 변환이 답변 근거 보존에 유리할 것으로 예상한다.

#### H3. Parent-Child 구조는 검색 정밀도와 문맥 보존을 동시에 개선할 것이다

작은 청크는 검색에는 유리하지만 조건이나 예외 문맥이 잘릴 수 있다.

작은 child 청크로 검색하고 원본 parent 문서를 함께 제공하면 정보 누락을 줄일 수 있을 것으로 예상한다.

#### H4. 변경 감지 기반 재수집은 최신성 충돌을 줄일 것이다

변경된 페이지만 교체하고 삭제된 문서를 관리하면 과거 정보와 최신 정보가 동시에 검색되는 문제를 줄일 수 있다.

#### H5. QA 평가셋은 파이프라인 개선 효과를 객관적으로 비교할 수 있게 할 것이다

단순 수집 건수나 파일 생성 여부가 아니라, 실제 사용자 질문에 필요한 근거가 검색되는지를 기준으로 데이터 품질을 평가할 수 있다.

## 4. Experiment Design

### 4.1 수행할 실험

#### E1. 수집 대상 페이지 검증

* 업무별 URL 수집 결과 확인
* 필수 페이지 누락 여부 점검
* 분석 필요 페이지의 수집 여부 재검토
* 중복 URL과 리다이렉트 URL 확인

#### E2. HTML → 텍스트 변환기 비교

다음 방식의 결과를 비교한다.

* BeautifulSoup 직접 파싱
* Trafilatura 등 범용 본문 추출기
* 사이트 구조 기반 커스텀 파서

평가 대상은 다음과 같다.

* 제목 및 소제목 보존
* 문단 순서 보존
* 표 구조 보존
* 링크 및 첨부파일 보존
* 불필요한 메뉴·푸터 제거
* 동일 입력에 대한 동일 출력 여부

#### E3. 청킹 방식 비교

* 글자 수 기반 단순 청킹
* 문단 기반 청킹
* 소제목 기반 구조 청킹
* 표 단위 청킹

질의-답변 평가셋을 사용하여 정답 근거 청크가 검색 가능한지 비교한다.

#### E4. 메타데이터 필터 효과 검증

다음 두 조건을 비교한다.

1. 전체 6개 업무를 대상으로 검색
2. `business_function` 필터 적용 후 검색

다른 업무의 문서가 검색 결과에 포함되는 비율을 비교한다.

#### E5. Parent-Child 데이터 구조 검증

* child 청크만 제공한 경우
* child 검색 후 parent 문맥을 함께 제공한 경우

조건, 예외, 신청 절차가 긴 질문에서 답변 근거 보존 정도를 비교한다.

#### E6. 변경 감지 및 재수집 검증

테스트 페이지 또는 저장된 HTML을 수정하여 다음 상태를 재현한다.

* 신규 페이지 추가
* 기존 페이지 본문 수정
* 첨부파일 추가
* 기존 페이지 삭제
* URL 유지 및 내용 변경

파이프라인이 각 상태를 올바르게 판별하는지 확인한다.

#### E7. 정성 오류 분석

수집 및 변환 실패 사례를 다음 유형으로 분류한다.

* 본문 영역 오탐지
* 메뉴 또는 푸터 혼입
* 표 구조 손실
* 링크 및 첨부파일 누락
* 이미지 정보 누락
* 동적 데이터 미수집
* 중복 페이지 적재
* 잘못된 업무 분류
* 변경 페이지 미감지

| 평가     | 수집 성공 여부     | QA 근거 검색 가능 여부        |

### 4.3 Evaluation Metrics 및 검증

#### 수집 범위 평가

* **URL Coverage**

  * 정의한 필수 URL 중 실제 수집에 성공한 비율

```text
수집 성공 필수 URL 수 / 전체 필수 URL 수
```

* **Collection Success Rate**

  * 전체 요청 URL 중 정상 HTML을 확보한 비율

* **Duplicate Rate**

  * 동일하거나 사실상 동일한 페이지가 중복 수집된 비율

#### 파싱 품질 평가

* 제목 보존율
* 소제목 보존율
* 표 보존율
* 링크 보존율
* 첨부파일 보존율
* 본문 순서 일치율
* 불필요 영역 제거율

전체 페이지 중 일부를 표본으로 선정하여 사람이 원본 HTML과 변환본을 비교한다.

#### 메타데이터 평가

* 필수 필드 누락률
* 업무 분류 정확도
* 하위 분류 정확도
* URL 및 문서 ID 연결 정확도
* 청크와 원본 문서 연결 정확도

#### QA 검색 평가

향후 기본 검색기를 연결할 수 있는 경우 다음 지표를 사용한다.

* **Hit@K**

  * 상위 K개 검색 결과 안에 정답 근거 문서가 포함되는 비율
* **MRR**

  * 정답 문서가 검색 결과에서 얼마나 높은 순위에 등장하는지 평가
* **Recall@K**

  * 필요한 근거 청크 중 상위 K개에서 검색된 비율
* **Source Accuracy**

  * 정답에 연결된 출처 URL이 올바른 비율

#### 갱신 파이프라인 평가

* 신규 페이지 감지 정확도
* 수정 페이지 감지 정확도
* 삭제 페이지 감지 정확도
* 변경되지 않은 페이지의 불필요한 재처리 비율
* 재수집 이후 최신 데이터 반영 여부

#### 사람 평가

무작위로 선정한 페이지를 대상으로 다음 항목을 3점 척도로 평가한다.

| 점수 | 기준                             |
| -- | ------------------------------ |
| 0  | 핵심 내용이 누락되거나 구조가 크게 손상됨        |
| 1  | 일부 내용은 활용 가능하지만 표·링크·문맥 손실이 있음 |
| 2  | 원문 의미와 구조가 대부분 보존되어 RAG에 활용 가능 |

## 9. 프로젝트 해석 및 최종 방향

우리 팀은 본 프로젝트를 다음과 같이 해석하였다.

> 이 프로젝트의 핵심은 웹페이지를 많이 긁어오는 것이 아니라, 예솜24가 정확한 답변을 생성할 수 있도록 **수집할 가치가 있는 데이터를 선별하고, 검색에 적합한 구조로 변환하며, 최신 상태를 유지할 수 있는 데이터 파이프라인을 설계하는 것**이다.

따라서 프로젝트 진행 시 다음 원칙을 따른다.

1. 수집 이전에 업무별 사용자 질문과 필요한 근거 페이지를 먼저 정의한다.
2. 원본 HTML과 출처 URL을 항상 보존한다.
3. 텍스트 변환 과정에서는 정보 누락과 임의 요약을 최소화한다.
4. 업무와 하위 분류 메타데이터를 통해 계층 검색이 가능하도록 한다.
5. 작은 청크와 원본 페이지의 관계를 유지한다.
6. 변경·삭제된 데이터가 검색 결과에 남지 않도록 갱신 구조를 설계한다.
7. 수집 결과는 실제 질의-답변 테스트셋을 통해 검증한다.

이 접근은 단순 크롤링 결과물을 만드는 것보다 초기 설계와 검수가 더 필요하다.

하지만 최종적으로는 향후 RAG 모델, 임베딩 모델, 검색 알고리즘이 변경되더라도 반복해서 활용할 수 있는 안정적인 데이터 자산을 구축할 수 있다는 장점이 있다.
